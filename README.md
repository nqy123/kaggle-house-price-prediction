# Kaggle 房价预测：用可解释特征和 OOF 融合刻画房屋价值

![Result Card](assets/result_card.png)

## 这个项目在简历里的定位

这个项目适合展示回归建模基本功：缺失值语义处理、价格分布变换、房屋结构特征、线性模型与树模型融合，以及对 RMSLE 指标的理解。它不像一个普通练习题，更像一个小型自动估价系统。

## 业务问题与评价指标

根据房屋面积、质量、地段、年代、装修、车库、地下室等信息预测 `SalePrice`，评价指标是 RMSLE。业务上相当于为房屋估值系统建立一个稳健的自动定价模型。

当前最佳为 `outputs/submission_best_blend.csv`，Kaggle Public Score `0.12304`，本地 OOF RMSE `0.106410`，约前 20% 水平。

## 我的整体解决思路

我把这个项目拆成五步：

1. **先理解数据**：看目标分布、字段类型、缺失模式、train/test 差异和指标特点。
2. **再做清洗**：把明显的脏值、哨兵值、语义缺失和泄露风险处理干净。
3. **重点做特征工程**：不是机械造列，而是把业务直觉、数据结构和模型能力连接起来。
4. **用 OOF 做验证**：每个模型都尽量通过交叉验证产生 OOF，避免只看一次线上提交。
5. **最后复盘取舍**：线上不涨的复杂方案不保留，最终只保留稳定、能解释、能复现的版本。

## EDA 与数据理解

- SalePrice 呈明显右偏分布，直接回归原价会被高价房过度影响。
- 房价由地段、面积、质量、年代共同决定，变量之间存在大量非线性交互。
- 缺失值不一定代表数据脏，例如无地下室、无车库、无泳池本身就是重要房屋属性。

## 数据清洗策略

- 对 `SalePrice` 做 `log1p` 变换，使训练目标更贴近 RMSLE，并降低豪宅异常值影响。
- 把语义缺失补成 `None` 或 0，例如 Garage、Basement、Pool 相关字段。
- 对偏态数值特征做变换，对类别特征做统一编码，减少模型对稀有类别的误判。

## 特征工程：不是堆字段，而是写入业务逻辑

| 特征设计 | 为什么这样构造 | 预期收益 |
| --- | --- | --- |
| 总面积特征 | 买房时最直接的价值锚点是可使用面积，而不是单独某一块面积。 | 提升模型对整体房屋规模的感知。 |
| 总卫浴/门廊/地下室 | 生活便利性和附加空间会显著影响价格。 | 补充原始字段未显式表达的舒适度。 |
| 房龄和翻新年龄 | 同样面积的房子，新旧程度和翻新情况会导致价格差异。 | 帮助模型识别折旧和翻新溢价。 |
| 质量 x 面积 | 高质量的大房子和普通大房子的价值不是线性差异。 | 让树模型和线性模型都能捕捉品质溢价。 |

这部分是整个项目最值得讲的地方。我的处理原则是：**先解释变量为什么可能影响目标，再决定把它做成比例、计数、交叉、分箱、rank、文本向量还是序列语义表示**。这样做出来的特征不是“玄学调参”，而是有业务含义、有验证闭环的建模资产。

## 模型选择：为什么用这些模型

- 线性/岭回归类模型负责捕捉平滑的价格趋势，对 log 价格很稳。
- LightGBM、XGBoost、CatBoost 负责学习非线性关系和复杂交互。
- 用 OOF 预测做非负权重融合，避免只凭一次 public leaderboard 选择模型。
- 最终保留多模型 OOF optimized blend。

我没有把模型当成黑箱堆叠，而是根据数据形态选择模型：结构化数据优先树模型，类别特征多时重视 CatBoost，高维匿名数值用 LightGBM，文本任务则先用 TF-IDF 建强 baseline，再用 Transformer 补语义理解。

## 验证与防止过拟合

- 使用 OOF 或交叉验证观察本地泛化表现。
- 区分 public/private 分数，避免过度追逐单次 public leaderboard。
- 严格排除 ID、target、后竞赛标签等泄露来源。
- 对融合方案做线上复盘：如果复杂方案不涨分，就回退到更稳定版本。

## 运行结果分析

- 这个项目的关键不是单模型，而是让不同模型分工：线性模型稳，树模型强，融合降低方差。
- Public `0.12304` 说明清洗和特征工程已经超过普通 baseline。
- 继续提升主要依赖异常房源处理和 Neighborhood 细分，而不是简单堆更多模型。

## 如何复现

安装依赖：

```bash
pip install -r requirements.txt
```

复现时先从 Kaggle 下载原始数据到项目约定的数据目录。部分仓库为了保持轻量，只保留最佳提交文件、实验日志和核心说明；如果仓库中存在 `src/`、`notebooks/` 或 `kaggle_kernel_*`，优先从这些入口运行训练。

常见入口示例：

```bash
python src/train_best.py
# 或在 Kaggle 上运行 kaggle_kernel_* 中的 GPU kernel
```

如果当前项目只保留了最佳产物，可直接查看 `outputs/` 中的 OOF、prediction、submission 和实验摘要文件。
